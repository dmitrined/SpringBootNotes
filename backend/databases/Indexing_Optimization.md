# Indexing & Optimization: Ускорение БД 🏎️

Индексы — это специальные структуры данных, которые позволяют базе данных быстро находить строки без полного сканирования таблицы (Full Table Scan).

---

## 1. Типы индексов 🧩

1. **B-Tree (Сбалансированное дерево):** Стандарт по умолчанию. Идеально для `=` и диапазонов (`BETWEEN`, `>`, `<`).
2. **Hash Index:** Сверхбыстрый поиск только по равенству (`=`). Не работает для диапазонов.
3. **GIN / GiST:** Для сложных данных (массивы, JSONB, полнотекстовый поиск).

---

## 2. Ключевые понятия 🧱

- **Primary Key (PK):** Уникальный индекс, по которому строки физически (в InnoDB) или логически упорядочены.
- **Composite Index (Составной):** Индекс по нескольким колонкам. *Важно:* Правило левого префикса — индекс `(A, B)` сработает для `WHERE A=...`, но не для `WHERE B=...`.
- **Unique Index:** Гарантирует уникальность значений.
- **Covering Index (Покрывающий):** Ситуация, когда все данные, запрошенные в `SELECT`, уже есть в самом индексе. БД даже не лезет в таблицу за данными.

---

## 3. Как понять, работает ли индекс? 🔎

Используйте команду **EXPLAIN**:
```sql
EXPLAIN ANALYZE SELECT * FROM users WHERE email = 'test@mail.com';
```

### На что смотреть в выводе:
- **Index Scan:** Хорошо (используем индекс).
- **Index Only Scan:** Отлично (покрывающий индекс).
- **Seq Scan (Sequential Scan):** Плохо для больших таблиц (сканируем всё подряд).
- **rows:** Примерное количество строк, которые БД планирует просмотреть.

---

## 4. Проблема N+1 в Hibernate/JPA 🐘

Это ситуация, когда для получения списка из `N` объектов с вложенными сущностями выполняется `1` запрос для списка и `N` отдельных запросов для каждого вложенного объекта.

### Решения:
1. **Join Fetch (JPQL):** `SELECT u FROM User u JOIN FETCH u.posts`.
2. **Entity Graph:** Декларативное описание полей для загрузки.
3. **Batch Size:** Загрузка связанных сущностей пачками, а не по одной.

---

## 5. Оптимизация запросов 🚀

1. **SELECT only what you need:** Не делайте `SELECT *`, если нужны 2 колонки.
2. **Avoid functions on indexed columns:** `WHERE YEAR(date) = 2023` убьет индекс. Используйте `WHERE date >= '2023-01-01' AND date < '2024-01-01'`.
3. **Analyze & Vacuum (в PostgreSQL):** БД нужна статистика для планировщика.
4. **Denormalization:** Иногда дублирование данных ускоряет чтение, избавляя от сложных JOIN-ов.
