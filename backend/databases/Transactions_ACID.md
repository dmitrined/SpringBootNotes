# Transactions & ACID: Надежность данных 🛡️

Транзакция — это логическая единица работы с базой данных, которая переводит её из одного согласованного состояния в другое.

---

## 1. Принципы ACID 🧩

| Буква | Название | Суть |
| :--- | :--- | :--- |
| **A** | **Atomicity** (Атомарность) | Либо выполняются все операции транзакции, либо ни одна ("всё или ничего"). |
| **C** | **Consistency** (Согласованность) | Транзакция не нарушает правила и ограничения БД (foreign keys, constraints). |
| **I** | **Isolation** (Изолированность) | Параллельные транзакции не должны влиять друг на друга до завершения. |
| **D** | **Durability** (Надежность) | Если транзакция закоммичена, её результат сохранится даже при сбое питания. |

---

## 2. Уровни изоляции транзакций 🧱

Чем выше уровень изоляции, тем меньше аномалий, но ниже производительность.

| Уровень | Dirty Read | Non-repeatable Read | Phantom Read |
| :--- | :---: | :---: | :---: |
| **Read Uncommitted** | ✅ Да | ✅ Да | ✅ Да |
| **Read Committed** | ❌ Нет | ✅ Да | ✅ Да |
| **Repeatable Read** | ❌ Нет | ❌ Нет | ✅ Да |
| **Serializable** | ❌ Нет | ❌ Нет | ❌ Нет |

### Определение аномалий:
- **Dirty Read (Грязное чтение):** Чтение данных, которые еще не закоммичены другой транзакцией.
- **Non-repeatable Read:** Повторное чтение той же строки дает другой результат (строку изменили и закоммитили).
- **Phantom Read (Фантомное чтение):** Повторный запрос с тем же условием возвращает другие строки (добавлены/удалены новые записи).

## 3. Транзакции в Spring (@Transactional) 🍃

Spring упрощает работу с транзакциями через AOP (прокси).

### Распространение (Propagation):
- **REQUIRED (по умолчанию):** Если есть транзакция — используем её, если нет — создаем новую.
- **REQUIRES_NEW:** Всегда создает новую транзакцию, приостанавливая текущую.
- **MANDATORY:** Требует наличие транзакции, иначе кидает исключение.
- **SUPPORTS:** Если есть транзакция — используем, если нет — выполняем без неё.

### Изоляция (Isolation):
Можно задать уровень прямо в аннотации:
`@Transactional(isolation = Isolation.REPEATABLE_READ)`

---

## 4. Блокировки (Locking) 🔒

1. **Optimistic Locking (Оптимистичная):** Проверяет версию записи перед обновлением (`@Version` в JPA). Если версия изменилась — кидает `OptimisticLockException`. Подходит для систем с редкими конфликтами.
2. **Pessimistic Locking (Пессимистичная):** Блокирует запись в БД сразу (`SELECT ... FOR UPDATE`). Никто другой не может её трогать, пока транзакция не завершится. Подходит для критических финансовых операций.
