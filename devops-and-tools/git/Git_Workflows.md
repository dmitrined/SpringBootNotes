# Git Workflows: Стратегии командной разработки 🌿

Выбор правильной стратегии ветвления определяет, насколько легко команде будет интегрировать код и выпускать релизы.

---

## 1. GitFlow 🌊

Классическая и самая структурированная модель. Подходит для проектов с четким циклом релизов.

### Основные ветки:
- **`master` / `main`:** Содержит только стабильный код, готовый к деплою в продакшн.
- **`develop`:** Главная ветка разработки. Сюда вливаются все фичи.
- **`feature/`:** Ветки для новых функций. Ответвляются от `develop`, вливаются обратно в `develop`.
- **`release/`:** Подготовка релиза (багфиксы перед выпуском). Ответвляется от `develop`, вливается в `main` и `develop`.
- **`hotfix/`:** Срочные правки багов на продакшене. Ответвляются от `main`, вливаются в `main` и `develop`.

---

## 2. GitHub Flow 🚀

Максимально простая модель. Идеально для Web-приложений и Continuous Delivery (CD).

### Принцип работы:
1. Любая ветка, ответвленная от `main`, должна быть готова к деплою.
2. Создаем ветку с понятным именем (например, `add-login-page`).
3. Регулярно пушим коммиты в удаленный репозиторий.
4. Открываем **Pull Request (PR)** для обсуждения и код-ревью.
5. После аппрува и прохождения тестов мержим в `main`.
6. Сразу деплоим `main` в продакшн.

---

## 3. Trunk-Based Development 🛤️

Разработчики работают в одной ветке (`trunk` / `main`) или очень короткоживущих ветках (не более дня).

- **Зачем?** Чтобы избежать "Merge Hell" (ада при слиянии огромных веток).
- **Как выкатывать недоделанные фичи?** Используются **Feature Toggles** (переключатели фич в коде). Фича мержится в `main`, но выключена для пользователей.

---

## 4. Слияние: Merge vs Rebase 🔀

| Метод | Что делает | Плюс | Минус |
| :--- | :--- | :--- | :--- |
| **Merge** | Создает новый коммит слияния. | Сохраняет историю как она была (нелинейно). | "Грязная" история с кучей merge-коммитов. |
| **Rebase** | Переносит ваши коммиты "поверх" целевой ветки. | Линейная, чистая история. | Переписывает историю (опасно для общих веток). |

> **Золотое правило:** Никогда не делайте `rebase` в публичных ветках (`main`, `develop`), над которыми работают другие люди. Используйте его только для своих локальных feature-веток.

---

## 5. Полезные советы для Middle 💡

1. **Пишите понятные коммиты:** Используйте [Conventional Commits](https://www.conventionalcommits.org/) (`feat: ...`, `fix: ...`, `docs: ...`).
2. **Используйте `git commit --amend`:** Если забыли добавить файл или опечатались в последнем коммите.
3. **`git cherry-pick`:** Если нужно перенести один конкретный коммит из одной ветки в другую без полного слияния.
4. **`git reflog`:** "Спасательный круг". Показывает историю всех действий (даже тех, что удалили коммиты через `reset --hard`).
